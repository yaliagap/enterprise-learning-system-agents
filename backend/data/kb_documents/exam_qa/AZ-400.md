## Skill Area: Configure processes and communications (10–15%)

### Q1
**Question:** A development team using Azure DevOps wants to automatically link work items (User Stories, Tasks) to commits and pull requests. What is the SIMPLEST way to achieve this in Azure Repos?

**Options:**
- A. Configure a webhook to update work items when a commit is pushed
- B. Include the work item ID in the commit message with the # prefix (e.g., "Fixes #1234")
- C. Manually link commits to work items in Azure Boards after each commit
- D. Use Azure Pipeline triggers to automatically link items at build time

**Correct Answer:** B  
**Explanation:** Azure DevOps automatically detects work item references in commit messages when the #ID syntax is used (e.g., "#1234" or "Fixes AB#1234"). This creates a traceable link from the commit/PR to the work item without any additional configuration. This feature is enabled by default in Azure Repos when integrated with Azure Boards.

### Q2
**Question:** A team wants to enforce that every pull request to the main branch requires approval from at least two code reviewers and all CI pipelines must pass before merging. Where is this configured in Azure DevOps?

**Options:**
- A. Azure Pipelines → Service Connections
- B. Azure Repos → Branch Policies on the main branch
- C. Azure Boards → Sprint Planning settings
- D. Azure Artifacts → Feed permissions

**Correct Answer:** B  
**Explanation:** Branch Policies in Azure Repos enforce rules on protected branches. Policies can require minimum reviewer count, prohibit self-approval, require linked work items, and require passing build validations (CI pipelines). Policies block direct pushes to protected branches and enforce review through PRs.

## Skill Area: Design and implement source control (15–20%)

### Q3
**Question:** A DevOps team uses GitHub for source control and wants a branching strategy that minimizes merge conflicts and supports continuous delivery where every commit to main is deployable. Which branching strategy aligns with these goals?

**Options:**
- A. Gitflow (feature branches, develop, release, hotfix branches)
- B. GitHub Flow (short-lived feature branches, PRs to main, immediate deploy)
- C. Trunk-Based Development with feature flags
- D. Both B and C — GitHub Flow and Trunk-Based Development are variants of the same CI/CD-friendly approach

**Correct Answer:** D  
**Explanation:** GitHub Flow and Trunk-Based Development both prioritize short-lived branches, frequent integration to a single mainline (main/trunk), and deployability of every commit. Gitflow's long-lived develop and release branches increase merge conflict risk and slow delivery. For CD pipelines, minimizing branch divergence is key.

### Q4
**Question:** A company stores Infrastructure as Code in Azure Repos and recently discovered that secrets (connection strings, API keys) were accidentally committed to the repository. What should the DevOps engineer do IMMEDIATELY and then PREVENT from happening again?

**Options:**
- A. Delete the commit from git history using `git filter-branch` and add `.gitignore` entries
- B. Rotate all exposed credentials, use `git filter-repo` to remove secrets from history, and implement pre-commit secret scanning with Microsoft Defender for DevOps or git-secrets hooks
- C. Make the repository private and restrict access
- D. Archive the repository and create a new one

**Correct Answer:** B  
**Explanation:** Rotating credentials invalidates exposed secrets. `git filter-repo` (preferred over deprecated `filter-branch`) rewrites history to remove secrets. Pre-commit hooks or Defender for DevOps (GitHub Advanced Security for Azure DevOps) scan for secret patterns before commits land in the remote, preventing future leaks. Making the repo private doesn't help if the secret is already compromised.

## Skill Area: Design and implement build and release pipelines (40–45%)

### Q5
**Question:** A multi-stage Azure Pipeline deploys an application to Dev, QA, and Production environments. The team requires that deployments to Production must be manually approved by two specific people AND can only occur between 6 PM and 8 AM on weekdays. How is this configured in Azure Pipelines YAML?

**Options:**
- A. Add approval checks and a business hours deployment schedule in the Pipeline YAML trigger section
- B. Configure Environment protection rules (approvals) and a time-window business hours check on the Production Environment resource in Azure DevOps
- C. Create a separate release pipeline with manual gates for Production only
- D. Use Pipeline variables to skip the deployment stage unless a manual variable is set

**Correct Answer:** B  
**Explanation:** Azure DevOps Environments support protection rules including manual approvals (with specific required approvers), exclusive locks, branch control checks, and business hours checks. YAML pipelines reference environments using the `environment:` keyword in deployment jobs — all checks defined on the environment apply automatically before the deployment job runs.

### Q6
**Question:** A team's Azure Pipeline builds a .NET application and currently takes 45 minutes to complete. Investigation shows that package restore takes 20 minutes. What is the BEST way to reduce build time?

**Options:**
- A. Upgrade to a larger self-hosted agent with more CPU cores
- B. Enable pipeline caching for the NuGet package cache directory keyed on the packages.lock.json hash
- C. Pre-install all NuGet packages on the agent image
- D. Move package restore to a separate parallel job

**Correct Answer:** B  
**Explanation:** Pipeline caching (using the `Cache` task) stores the NuGet packages directory between runs keyed by the lock file hash. If dependencies haven't changed, the cache is restored and `dotnet restore` runs in seconds instead of downloading packages. This is the standard optimization for dependency installation in CI pipelines.

### Q7
**Question:** A DevOps engineer is designing a deployment strategy for a high-traffic API where they need to test a new version with 10% of production traffic before full rollout, with automatic rollback if the error rate exceeds 2%. Which deployment strategy supports this?

**Options:**
- A. Blue-Green deployment
- B. Canary deployment
- C. Rolling deployment
- D. Recreate deployment

**Correct Answer:** B  
**Explanation:** Canary deployments route a small percentage of traffic (e.g., 10%) to the new version while the rest continues on the stable version. Metrics are monitored (error rate, latency) and the rollout proceeds or rolls back based on thresholds. Azure Deployment Environments, Kubernetes with traffic splitting, or Azure Front Door weighted routing can implement canary. Blue-Green switches ALL traffic at once.

### Q8
**Question:** A GitHub Actions workflow builds a container image and pushes it to Azure Container Registry (ACR). The workflow runs on GitHub-hosted runners. What is the RECOMMENDED authentication method to push images to ACR from GitHub Actions?

**Options:**
- A. Store the ACR admin username and password in GitHub Secrets
- B. Use a service principal with a client secret stored in GitHub Secrets
- C. Use OpenID Connect (OIDC) federation between GitHub and Azure — workload identity federation
- D. Generate a long-lived SAS token for ACR

**Correct Answer:** C  
**Explanation:** Workload Identity Federation (OIDC) allows GitHub Actions to authenticate to Azure without storing any secrets in GitHub. GitHub generates an OIDC token for each workflow run, and Azure AD validates it against a registered federated credential on a service principal or managed identity. This eliminates secret rotation and secret leak risks — the current best practice for CI/CD to Azure.

### Q9
**Question:** An organization uses Azure Pipelines and wants to publish NuGet packages to be shared across 15 development teams. Packages should only be promoted to a "release" quality gate after passing automated tests. Which Azure DevOps service and feature handles this?

**Options:**
- A. Azure Repos with package-per-branch tagging
- B. Azure Artifacts with feed views (Prerelease and Release views)
- C. Azure Container Registry with image tagging conventions
- D. GitHub Packages with version pinning

**Correct Answer:** B  
**Explanation:** Azure Artifacts hosts NuGet, npm, Maven, and Python packages. Feed views (e.g., @prerelease, @release) act as promotion gates. Packages are published to @prerelease after build, promoted to @release after test validation. Teams consuming packages can pin to @release to receive only quality-validated versions. Pipeline tasks handle promotion automatically.

## Skill Area: Develop a security and compliance plan (10–15%)

### Q10
**Question:** A DevOps team wants to automatically scan container images in their Azure Pipelines CI workflow for known CVEs before pushing to Azure Container Registry. Which tool integrates natively with Azure DevOps pipelines for this purpose?

**Options:**
- A. Azure Policy Guest Configuration
- B. Microsoft Defender for Containers with Trivy or Qualys scanner tasks in the pipeline
- C. Azure Security Center adaptive application controls
- D. Azure Blueprints security controls

**Correct Answer:** B  
**Explanation:** Microsoft Defender for Containers includes vulnerability assessment for container images. In pipelines, open-source tools like Trivy (commonly used in Azure Pipelines and GitHub Actions tasks) scan images for CVEs. GitHub Advanced Security for Azure DevOps (GHAS) also provides dependency scanning and secret scanning. The pipeline task fails the build if critical vulnerabilities are found.

### Q11
**Question:** An organization's compliance policy requires that all Infrastructure as Code (Bicep/ARM templates) be scanned for security misconfigurations before deployment. Which approach integrates directly into an Azure Pipeline?

**Options:**
- A. Run Azure Policy evaluation on the target subscription before deployment
- B. Use `az deployment what-if` to preview changes only
- C. Integrate a static analysis tool (e.g., Checkov, PSRule for Azure, or Microsoft Security DevOps extension) as a pipeline step before the deployment step
- D. Enable Azure Defender for Resource Manager after deployment

**Correct Answer:** C  
**Explanation:** Static analysis tools like Checkov, PSRule for Azure, or the Microsoft Security DevOps Azure DevOps extension scan IaC files for security misconfigurations (open NSG rules, encryption disabled, public storage accounts) before deployment, enabling "shift left" security. `what-if` previews changes but doesn't evaluate security posture.

## Skill Area: Implement an instrumentation strategy (10–15%)

### Q12
**Question:** A DevOps team needs to implement distributed tracing across a microservices application that spans Azure App Service, Azure Functions, and Azure Service Bus. They want a single transaction map showing all service dependencies. Which Azure Monitor feature provides this visualization?

**Options:**
- A. Azure Monitor Workbooks
- B. Application Insights — Application Map
- C. Azure Monitor Metrics Explorer
- D. Log Analytics query dashboard

**Correct Answer:** B  
**Explanation:** Application Insights Application Map automatically discovers and visualizes all components of a distributed application (services, dependencies, queues, databases) and shows request rates, failure rates, and latency between components. It requires each service to be instrumented with the Application Insights SDK or auto-instrumentation, and uses the same Instrumentation Key or connection string to correlate traces.

### Q13
**Question:** A DevOps engineer needs to create an alert that fires when the number of failed requests in a specific Application Insights resource exceeds 50 in any 5-minute window. Which alert type should be used?

**Options:**
- A. Azure Monitor Metric Alert on the failed requests metric
- B. Azure Monitor Log Alert with a KQL query counting failed requests in the last 5 minutes
- C. Application Insights Smart Detection (Adaptive Anomaly Detection)
- D. Azure Service Health Alert

**Correct Answer:** B  
**Explanation:** Application Insights failed request counts are available as both metrics and logs. A Log Alert with a KQL query (`requests | where success == false | where timestamp > ago(5m) | count`) with a threshold of 50 provides the flexibility to filter by specific operation names, response codes, or other attributes. Metric Alerts work for aggregate counts but offer less filtering granularity.
