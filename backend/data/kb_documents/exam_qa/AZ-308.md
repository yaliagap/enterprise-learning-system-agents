## Skill Area: Design AI platform architecture and governance (30–35%)

### Q1
**Question:** An enterprise is building an internal AI platform for 500 developers across 8 business units. Each business unit needs isolated AI project workspaces with their own budgets, data access controls, and AI model deployments, while sharing common infrastructure (Azure OpenAI PTU deployments, network connectivity, monitoring). Which Azure AI Foundry topology addresses this?

**Options:**
- A. One Azure AI Foundry Hub per business unit, with no shared infrastructure
- B. A hub-and-spoke topology: one shared Azure AI Foundry Hub (shared OpenAI deployments, networking, monitoring) with one Project per team/business unit, inheriting hub resources with isolated RBAC
- C. A single Azure AI Foundry Project shared by all business units with folder-based isolation
- D. Separate Azure subscriptions per business unit with independent AI resource deployments

**Correct Answer:** B  
**Explanation:** Azure AI Foundry's Hub-Project model is designed for exactly this scenario. The Hub holds shared infrastructure: Azure OpenAI Service (expensive Provisioned Throughput Unit deployments amortized across teams), Azure AI Search indexes, Storage, Key Vault, and managed VNet. Projects inherit hub resources but have isolated RBAC, separate experiment tracking, and independent deployment environments. This balances cost efficiency with governance isolation.

### Q2
**Question:** A solutions architect is designing governance controls for an enterprise AI platform. The policy requires that all AI model deployments must be logged, developers cannot deploy unapproved models, and all AI-generated content must pass content safety filtering. How should these controls be architected?

**Options:**
- A. Rely on developers following documented guidelines and periodic audit reviews
- B. Azure Policy to restrict approved model deployments to an allowlist of model IDs; Azure AI Content Safety as a mandatory gateway; Azure Monitor/Diagnostic settings for audit logging of all inference calls
- C. Microsoft Defender for Cloud AI recommendations as the sole governance mechanism
- D. Azure Blueprints to deploy AI resources with locked configurations

**Correct Answer:** B  
**Explanation:** Enterprise AI governance requires policy enforcement at multiple layers: (1) Azure Policy restricts which AI model IDs can be deployed (allowlist) and enforces configuration (content filters enabled, private endpoints required); (2) Azure AI Content Safety as a mandatory API gateway for all inference traffic; (3) Diagnostic settings on Azure OpenAI resources route all inference logs to Log Analytics for audit trails. This is a defense-in-depth governance architecture.

### Q3
**Question:** An architect is designing the RBAC model for an Azure AI Foundry platform. Data scientists need to train and deploy models. ML engineers need to manage pipelines but not modify network settings. Security engineers need read-only access to audit logs and configurations. Which approach aligns with the principle of least privilege?

**Options:**
- A. Grant all users the Contributor role on the Azure AI Foundry Hub resource group
- B. Create custom RBAC roles scoped to specific Azure AI Foundry operations: AI Developer role for data scientists, ML Engineer role for pipeline management, Reader + Diagnostic Settings Reader for security engineers — assigned at Project scope, not Hub scope
- C. Use a single shared service principal for all platform access
- D. Grant data scientists Owner role on their Project only

**Correct Answer:** B  
**Explanation:** Custom RBAC roles with specific Azure AI Foundry data plane permissions (Microsoft.MachineLearningServices/workspaces/*) enable precise access control. Scoping at Project level prevents inadvertent access to other projects sharing the hub. Built-in roles like Azure AI Developer provide a reasonable baseline. The Owner role grants excessive access including RBAC management — never assigned to data scientists by default.

## Skill Area: Design data and knowledge architecture for AI (25–30%)

### Q4
**Question:** An enterprise needs to architect a knowledge base for an AI assistant that covers 10 million documents updated continuously, requires sub-second query latency for semantic search, and must support hybrid search (keyword + vector). Which vector store architecture is recommended?

**Options:**
- A. Azure Blob Storage with in-memory FAISS for vector search
- B. Azure AI Search with HNSW vector index + hybrid (BM25 keyword + vector) + semantic ranking as the reference architecture for enterprise RAG at scale
- C. Azure Cosmos DB for NoSQL with custom vector serialization
- D. Azure Cache for Redis with RediSearch module

**Correct Answer:** B  
**Explanation:** Azure AI Search is the Microsoft-recommended vector store for enterprise RAG. It provides: HNSW indexing for sub-second ANN search at scale, native hybrid search combining BM25 (keyword) and vector, L2 semantic re-ranking for improved relevance, and incremental indexing for continuous document updates via Change Data Capture (CDC) patterns with Azure Data Factory or Event Grid triggers.

### Q5
**Question:** An AI platform architect needs to design a feature store for a recommendation system. Features (user embeddings, product attributes, real-time behavioral signals) must be available at low latency for inference AND for offline model training with historical consistency. Which Azure architecture pattern addresses this?

**Options:**
- A. Azure SQL Database as both the online and offline store
- B. Online feature store (Azure Cache for Redis for sub-millisecond inference) + offline feature store (Azure Data Lake Storage Gen2 with Delta Lake for training consistency) + Azure ML Feature Store to manage feature definitions and lineage
- C. Azure Cosmos DB as a single store for both online and offline use cases
- D. Azure Synapse Analytics for both real-time serving and offline training

**Correct Answer:** B  
**Explanation:** The Lambda architecture for feature stores separates concerns: Redis serves online inference with microsecond latency; ADLS Gen2 with Delta Lake provides point-in-time correct historical features for training (preventing data leakage). Azure ML Feature Store (preview) manages feature definitions, transformation code, and lineage between online and offline stores. This is the enterprise ML feature store reference architecture.

## Skill Area: Design AI security and compliance architecture (20–25%)

### Q6
**Question:** An enterprise deploys Azure OpenAI Service and requires that ALL inference traffic must remain within the corporate network — no data should traverse the public internet. What network architecture achieves this?

**Options:**
- A. Configure Azure OpenAI with API key authentication and use HTTPS for encryption in transit
- B. Deploy Azure OpenAI Service with public network access disabled, create a Private Endpoint in the corporate VNet, and route all AI traffic through the Private Endpoint using private DNS zone resolution
- C. Use Azure VPN Gateway to route all traffic to Azure OpenAI public endpoints through the corporate network
- D. Deploy Azure OpenAI in a dedicated Azure region with higher isolation

**Correct Answer:** B  
**Explanation:** Azure OpenAI Service supports Private Endpoints that assign a private IP address from a VNet to the OpenAI resource. With public access disabled, all API calls must originate from within the VNet or connected networks (ExpressRoute/VPN). A Private DNS Zone (privatelink.openai.azure.com) resolves the OpenAI endpoint hostname to the private IP. This ensures no inference data leaves the private network boundary.

### Q7
**Question:** An architect is designing data classification and access controls for an AI platform that processes documents containing PII, financial data, and public information. The AI models must only access documents at the appropriate sensitivity level based on the requesting user's clearance. Which pattern implements this?

**Options:**
- A. Separate Azure AI Search indexes per sensitivity level with RBAC-controlled access at the application layer — users with higher clearance can query higher-sensitivity indexes
- B. A single Azure AI Search index with all documents, relying on prompt instructions to not return sensitive data
- C. Encrypt all documents with the same key and decrypt at query time for all users
- D. Use Azure Information Protection labels as metadata only, without access enforcement

**Correct Answer:** A  
**Explanation:** Security trimming in RAG requires document-level or index-level access control. The recommended architecture uses separate indexes per classification level (public, internal, confidential, highly confidential) with RBAC controlling which application identities (managed identities per user role or application tier) can query each index. Row-level security (document-level ACLs on a single index) is also supported in Azure AI Search via security filters.

## Skill Area: Design AI cost and operational architecture (15–20%)

### Q8
**Question:** An enterprise AI platform processes variable token volumes: 50M tokens/day on weekdays and 5M tokens/day on weekends. The team wants to minimize cost while ensuring availability. Which Azure OpenAI capacity model is recommended?

**Options:**
- A. Provisioned Throughput Units (PTU) sized for peak weekday volume
- B. Pay-as-you-go (PAYG) token-based pricing for all traffic
- C. PTU for baseline weekday load (30M tokens/day) + PAYG overflow for peak bursts and weekend traffic
- D. Standard tier with a fixed monthly commitment

**Correct Answer:** C  
**Explanation:** The hybrid PTU + PAYG model is the Microsoft-recommended cost optimization strategy for variable AI workloads. PTU provides guaranteed throughput at a lower per-token cost (committed capacity) for predictable baseline load. PAYG handles bursts and low-volume periods without over-provisioning PTU for 50M token/day peaks that only occur on weekdays. PTU is billed hourly regardless of usage, so over-provisioning wastes money on weekends.

### Q9
**Question:** An architect needs to implement AI cost governance across 10 teams sharing a single Azure OpenAI resource. Each team should have a monthly token budget and receive alerts when they approach 80% of their budget. How is this implemented?

**Options:**
- A. Azure Cost Management budget alerts on the Azure OpenAI resource for the total subscription cost
- B. Azure API Management (APIM) as a gateway with per-subscription token counting policies and Azure Monitor alerts based on APIM token metrics per team API subscription key
- C. Manual monthly review of Azure OpenAI token usage logs
- D. Azure Policy to restrict token usage per team

**Correct Answer:** B  
**Explanation:** Azure API Management as an AI gateway enables per-team token governance: (1) Each team uses a separate APIM subscription key; (2) APIM policy captures token counts from Azure OpenAI response headers (x-ratelimit-remaining-tokens); (3) Custom metrics in Azure Monitor track cumulative tokens per subscription key; (4) Metric Alerts fire when a team reaches 80% of their monthly budget. This is the recommended AI cost chargeback and governance architecture pattern.
