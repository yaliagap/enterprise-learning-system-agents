## Skill Area: Design and implement an MLOps infrastructure (20–25%)

### Q1
**Question:** An ML team wants to define their Azure Machine Learning training pipeline as code (IaC) so that the entire infrastructure (workspace, compute clusters, data stores) can be reproduced in a new region in case of disaster. Which Azure technology is recommended for this?

**Options:**
- A. Terraform or Bicep templates checked into version control, deployed via Azure Pipelines
- B. Manual Azure portal provisioning documented in a wiki
- C. Azure ML Python SDK v2 scripts for workspace creation
- D. ARM templates exported from the portal after initial setup

**Correct Answer:** A  
**Explanation:** Infrastructure as Code (Bicep or Terraform) checked into version control enables reproducible, auditable, and automated infrastructure provisioning. Azure Pipelines or GitHub Actions can deploy the AML workspace, compute clusters, and connected resources automatically. This enables disaster recovery (redeploy to another region) and environment consistency (dev/staging/prod parity). The Azure ML Python SDK v2 is for ML operations, not infrastructure provisioning.

### Q2
**Question:** An MLOps engineer is setting up a CI/CD pipeline for machine learning. When a data scientist pushes a change to a training script in GitHub, the pipeline should automatically run data validation, train the model, evaluate performance, register the model if metrics improve, and deploy to a staging endpoint. Which Azure services are used for this orchestration?

**Options:**
- A. Azure Synapse Analytics pipeline + Azure Functions
- B. GitHub Actions (or Azure Pipelines) → Azure Machine Learning pipeline jobs via the Azure ML CLI v2 or Python SDK v2
- C. Azure Data Factory + Azure Databricks jobs
- D. Azure Logic Apps with Azure ML connector

**Correct Answer:** B  
**Explanation:** The standard MLOps CI/CD pattern: GitHub Actions (or Azure Pipelines) triggers on code push; the workflow calls `az ml job create` (CLI v2) or the Azure ML Python SDK to submit a pipeline job to AML. The AML pipeline executes: data validation component → training component → evaluation component → conditional model registration. If metrics pass threshold, a deployment step updates the staging endpoint.

## Skill Area: Implement machine learning model lifecycle and operations (25–30%)

### Q3
**Question:** A data science team uses Azure Machine Learning to train multiple model versions over time. They need to track experiment metrics (accuracy, F1, RMSE), log parameters (learning rate, batch size), and artifacts (model files, confusion matrices) in a central location for comparison. Which Azure ML feature provides this?

**Options:**
- A. Azure ML Datasets
- B. MLflow Tracking integrated with Azure Machine Learning (experiments, runs, metrics, artifacts)
- C. Azure Monitor custom metrics
- D. Azure ML Environments

**Correct Answer:** B  
**Explanation:** Azure Machine Learning natively integrates MLflow as the experiment tracking backend. Using `mlflow.log_metric()`, `mlflow.log_param()`, and `mlflow.log_artifact()`, training runs automatically log to Azure ML experiments. The Azure ML Studio UI shows run comparison, metric charts, and artifact downloads. MLflow is the open standard supported by Azure ML, Databricks, and other ML platforms.

### Q4
**Question:** An MLOps engineer needs to package a trained scikit-learn model with its dependencies and register it in Azure ML so it can be deployed to multiple environments (real-time endpoint, batch endpoint, Azure Functions) without reimaging. What is the standard packaging approach?

**Options:**
- A. Save the model as a .pkl file and upload to Azure Blob Storage
- B. Register the model in the Azure ML Model Registry with an MLflow model format (model flavor, conda.yaml dependencies, and MLmodel specification)
- C. Package the model in a Docker container with all dependencies baked in
- D. Export the model as ONNX format only

**Correct Answer:** B  
**Explanation:** Registering models in the Azure ML Model Registry in MLflow format provides: portability across deployment targets (managed online endpoints, batch endpoints, Azure Functions), dependency specification via conda.yaml, model flavor metadata (sklearn, pytorch, tensorflow), and versioning with lineage tracking back to the training run. Azure ML managed online endpoints can deploy MLflow models without custom scoring scripts (no-code deployment).

### Q5
**Question:** An online prediction endpoint in Azure ML is receiving production traffic. The MLOps team needs to test a new model version with a small percentage of traffic (5%) before full rollout, while keeping the current model version serving 95% of requests. Which Azure ML deployment feature supports this?

**Options:**
- A. Creating a new endpoint and updating DNS routing
- B. Blue-green deployment by swapping endpoint deployments
- C. Traffic splitting across multiple deployments within a single Azure ML managed online endpoint
- D. Azure Traffic Manager weighted routing between two endpoints

**Correct Answer:** C  
**Explanation:** Azure ML managed online endpoints support multiple named deployments (e.g., "blue" and "green") with configurable traffic weights (e.g., blue=95%, green=5%). Traffic is split at the endpoint level. After validation, traffic can be gradually shifted (5% → 25% → 50% → 100%) without downtime or DNS changes. This is the native canary deployment mechanism in Azure ML endpoints.

## Skill Area: Design and implement a GenAIOps infrastructure (20–25%)

### Q6
**Question:** An ML engineering team is deploying a RAG application on Azure AI Foundry. They need a pipeline to automatically evaluate the quality of RAG responses daily on a set of 500 test questions, measuring groundedness, relevance, and coherence. Which Azure AI Foundry capability provides this?

**Options:**
- A. Azure ML AutoML evaluation runs
- B. Azure AI Foundry Evaluation runs using built-in AI-assisted quality metrics (groundedness, relevance, coherence, fluency)
- C. Custom KQL queries in Azure Monitor against inference logs
- D. Manual review by the team each day

**Correct Answer:** B  
**Explanation:** Azure AI Foundry provides built-in evaluation metrics for RAG applications: groundedness (is the answer supported by the retrieved context?), relevance (does the answer address the question?), coherence (is the answer well-structured?), and fluency (is the language natural?). These AI-assisted metrics use an LLM-as-judge pattern. Evaluation runs can be scheduled and integrated into CI/CD pipelines to gate deployments on quality thresholds.

### Q7
**Question:** An MLOps engineer notices that a deployed GPT-4o deployment on Azure OpenAI Service is experiencing token throughput limits, causing 429 errors during peak hours. The team wants to handle these gracefully. What is the RECOMMENDED architecture pattern?

**Options:**
- A. Increase the retry count in the application to 20 retries with no delay
- B. Implement exponential backoff with jitter on 429 responses, add a queue-based load leveling pattern (Azure Service Bus) for non-real-time requests, and consider PTU deployments for predictable throughput
- C. Provision a second Azure OpenAI deployment in the same region to double throughput
- D. Switch to a smaller model (GPT-3.5-Turbo) to reduce latency

**Correct Answer:** B  
**Explanation:** The complete resilience pattern for Azure OpenAI includes: (1) Exponential backoff with jitter for synchronous retries on transient 429s; (2) Queue-based load leveling using Azure Service Bus for batch/async workloads — decouple producers from Azure OpenAI calls and process at a sustainable rate; (3) PTU (Provisioned Throughput Units) for predictable throughput guarantees on sustained high-volume workloads. This is the Azure OpenAI production resilience architecture.

## Skill Area: Implement generative AI quality assurance and observability (15–20%)

### Q8
**Question:** An MLOps team needs to detect when a deployed language model's output quality degrades over time (e.g., responses become less grounded, coherence drops). What is the monitoring strategy for a production RAG system?

**Options:**
- A. Monitor infrastructure metrics (CPU, memory, latency) only
- B. Implement LLM observability: log all input/output pairs, run scheduled AI-assisted evaluation jobs on a sample (groundedness, coherence, relevance scores), track score trends in Azure Monitor, and alert on threshold drops
- C. Monitor token consumption as a proxy for quality
- D. Run weekly manual human evaluation of 20 random responses

**Correct Answer:** B  
**Explanation:** LLM observability goes beyond infrastructure monitoring. The recommended pattern: (1) Log all inference calls (prompt, completion, latency, token count) to Log Analytics; (2) Schedule Azure AI Foundry evaluation runs on a random sample (e.g., 1%) of production queries; (3) Track AI quality metric trends (groundedness score, coherence score) as custom metrics in Azure Monitor; (4) Create alerts when metric moving averages drop below baseline. This detects model degradation, content drift, and RAG retrieval failures.

## Skill Area: Optimize generative AI systems and model performance (10–15%)

### Q9
**Question:** A language model deployed for code generation is producing responses that are technically correct but don't follow the company's specific coding conventions (naming, docstring format, error handling patterns). The team has 5,000 examples of correct code in the company's style. Which optimization technique best addresses this?

**Options:**
- A. Increase the temperature parameter to generate more varied responses
- B. Fine-tune the base model on the company's 5,000 code examples using Azure AI Foundry fine-tuning
- C. Add detailed style guidelines to every prompt (few-shot examples)
- D. Deploy a different foundation model

**Correct Answer:** B  
**Explanation:** Fine-tuning on company-specific examples teaches the model the target style, reducing the need for lengthy few-shot examples in every prompt (saving tokens). Azure AI Foundry supports supervised fine-tuning for GPT-4o-mini and other models with training data in JSONL format. Fine-tuning is the right choice for style/format consistency where the examples are high quality and the task is well-defined.
