## Skill Area: Implement and manage a data engineering solution (35–40%)

### Q1
**Question:** A data engineering team is migrating from Azure Synapse Analytics to Microsoft Fabric. They need to understand the equivalent of Synapse's "dedicated SQL pool" for large-scale structured data warehousing in Fabric. What is the correct Fabric item?

**Options:**
- A. Microsoft Fabric Lakehouse
- B. Microsoft Fabric Warehouse (SQL-based data warehouse on OneLake)
- C. Microsoft Fabric KQL Database
- D. Microsoft Fabric Eventstream

**Correct Answer:** B  
**Explanation:** Microsoft Fabric Warehouse is the SQL-based data warehousing experience in Fabric, equivalent to Azure Synapse dedicated SQL pools. It supports T-SQL querying, full DDL/DML, stored procedures, and views. Data is stored in OneLake in Delta Lake format. The Lakehouse is a data engineering artifact for data transformation and exploration using Spark notebooks and SQL analytics endpoint — it's not a traditional data warehouse.

### Q2
**Question:** A Fabric workspace has a Lakehouse with millions of Delta tables containing raw and processed data. A data engineer wants to enable SQL users (using tools like Power BI, SSMS, or Tableau) to query this data using standard T-SQL without moving the data. Which Fabric feature enables this?

**Options:**
- A. Export data to Azure SQL Database for SQL querying
- B. SQL Analytics Endpoint — the auto-generated read-only T-SQL endpoint for every Lakehouse
- C. Create a separate Fabric Warehouse and copy all Lakehouse data daily
- D. Azure Synapse Link for Cosmos DB

**Correct Answer:** B  
**Explanation:** Every Microsoft Fabric Lakehouse automatically generates a SQL Analytics Endpoint — a fully managed, read-only T-SQL interface over the Delta tables in the Lakehouse's OneLake storage. External BI tools and SQL clients can connect using the SQL connection string and query Delta tables using standard T-SQL without any data movement or duplication.

### Q3
**Question:** A data engineer needs to create a Fabric data pipeline that: reads raw JSON files from ADLS Gen2, applies data quality checks, transforms using Python/PySpark, and writes the result as Delta tables to a Fabric Lakehouse. Which Fabric items should be used?

**Options:**
- A. Fabric Data Pipeline + Fabric Notebook (PySpark)
- B. Power BI dataflow Gen2 only
- C. Fabric Eventstream for JSON ingestion
- D. KQL Queryset for transformation

**Correct Answer:** A  
**Explanation:** Fabric Data Pipelines (based on Azure Data Factory) orchestrate data movement and trigger Fabric Notebooks. Notebooks with PySpark process raw JSON, apply transformations and quality checks using Delta Lake operations, and write results as Delta tables to the Lakehouse. This is the standard ELT pattern in Microsoft Fabric for batch data processing.

## Skill Area: Ingest and transform data (30–35%)

### Q4
**Question:** A data team uses Microsoft Fabric and wants to create reusable data transformation logic with a visual interface (no-code/low-code) that automatically refreshes linked tables in a Fabric Lakehouse. Which Fabric item provides this?

**Options:**
- A. Fabric Notebook with scheduled triggers
- B. Fabric Dataflow Gen2 (Power Query-based transformations with OneLake output)
- C. Fabric Data Pipeline with copy activity
- D. Azure Data Factory dataflow

**Correct Answer:** B  
**Explanation:** Fabric Dataflow Gen2 uses Power Query M language and provides a visual transformation editor. Data engineers and analysts without Spark expertise can build transformation logic visually. Dataflows Gen2 can output directly to Fabric Lakehouses, Warehouses, and other destinations in OneLake. They replace the need for Python/Spark for many common transformation scenarios.

### Q5
**Question:** A retail company streams point-of-sale transaction data at 100,000 events per second into Microsoft Fabric for real-time inventory monitoring. Which Fabric service ingests this high-throughput event stream?

**Options:**
- A. Fabric Data Pipeline with a polling connector
- B. Fabric Eventstream — connected to Azure Event Hubs or Kafka, routing events to Fabric KQL Database (Eventhouse) for real-time analytics
- C. Fabric Dataflow Gen2 with HTTP connector
- D. OneLake Shortcuts pointing to the source system

**Correct Answer:** B  
**Explanation:** Microsoft Fabric Eventstream connects to real-time event sources (Azure Event Hubs, Kafka, IoT Hub, Custom App) and routes streaming data to Fabric destinations: KQL Database (Eventhouse) for real-time query, Lakehouse Delta tables for batch landing, or another Eventstream for fan-out. Eventhouse + KQL provides sub-second analytical queries over streaming data — the Fabric real-time intelligence solution.

### Q6
**Question:** A data engineer needs to join a Fabric Lakehouse Delta table (2 TB, daily batch updated) with a real-time stream from Eventstream for inventory alerts. Which Fabric capability enables joining batch and streaming data?

**Options:**
- A. Fabric Notebook with SparkStreaming + Delta table join
- B. KQL Database with materialized views joining Eventhouse streaming data with OneLake shortcuts to the Lakehouse Delta table
- C. Power BI DirectQuery over both sources simultaneously
- D. Fabric Data Pipeline with a merge activity

**Correct Answer:** B  
**Explanation:** In Fabric's real-time intelligence stack, KQL Databases (Eventhouses) can access OneLake Delta tables via Shortcuts and join them with streaming data in KQL queries. Alternatively, Fabric Notebooks with Structured Streaming can join a Spark Streaming source (from Eventhouse) with a static Delta table in the Lakehouse using a streaming join pattern. Both are valid; option B reflects the Fabric-native real-time analytics approach.

## Skill Area: Monitor and optimize data engineering solutions (15–20%)

### Q7
**Question:** A Fabric Spark notebook job that processes daily transaction data takes 4 hours to complete. After profiling, the team discovers that 90% of time is spent on a shuffle operation when joining two large tables on a low-cardinality key. What optimization should be applied?

**Options:**
- A. Increase the Spark cluster node count
- B. Use a broadcast join hint for the smaller table if it fits in memory; otherwise, optimize the partition key to reduce shuffle size
- C. Move the processing to Fabric Dataflow Gen2
- D. Switch from PySpark to KQL for the join

**Correct Answer:** B  
**Explanation:** Shuffle-heavy joins on low-cardinality keys cause data skew and excessive network I/O. Solutions: (1) If one table is small enough (typically < 1–2 GB), use `broadcast()` hint to broadcast it to all executors, eliminating shuffle entirely; (2) For large-large joins, repartition both tables on the join key with appropriate partition count before the join; (3) Use Delta Lake Z-ordering on join columns to improve data layout. These are the standard Spark join optimization techniques.

## Skill Area: Implement security for data engineering (10–15%)

### Q8
**Question:** A data engineer needs to ensure that sensitive columns (SSN, salary) in a Fabric Warehouse table are visible only to HR users, while other business users see masked values. Which Microsoft Fabric security feature handles column-level data masking?

**Options:**
- A. Azure Information Protection sensitivity labels on the Fabric item
- B. Dynamic Data Masking (DDM) on the Fabric Warehouse table — masks column values in query results for non-privileged users
- C. Row-Level Security (RLS) policies
- D. Fabric workspace-level RBAC role assignment

**Correct Answer:** B  
**Explanation:** Fabric Warehouse supports Dynamic Data Masking (inherited from SQL Server/Azure SQL Database). Masked columns show obfuscated values (e.g., "XXXX-XX-XXXX" for SSN) to users without the UNMASK permission. Users granted UNMASK (e.g., HR admin role) see the actual values. Row-Level Security controls which rows are visible; DDM controls which column values are readable.
