## Skill Area: Develop containerized solutions on Azure (20–25%)

### Q1
**Question:** A developer needs to containerize a Python FastAPI application and deploy it to Azure with the ability to scale to zero (no running containers when idle) and scale out automatically based on HTTP request load. Which Azure service should be used?

**Options:**
- A. Azure Kubernetes Service (AKS) with KEDA
- B. Azure Container Apps with HTTP scaling rules
- C. Azure Container Instances (ACI) with a load balancer
- D. Azure App Service with Docker container

**Correct Answer:** B  
**Explanation:** Azure Container Apps natively supports scale-to-zero and HTTP-triggered autoscaling without managing Kubernetes clusters. It uses KEDA (Kubernetes Event-Driven Autoscaling) under the hood. ACI doesn't auto-scale. App Service can run containers but doesn't scale to zero on the basic and standard tiers. AKS requires Kubernetes cluster management.

### Q2
**Question:** A developer is building a multi-container application with a web frontend, API backend, and Redis cache. They need to define the entire application stack in a configuration file and deploy it to Azure Container Apps as a single unit. What is the RECOMMENDED approach?

**Options:**
- A. Deploy each container as a separate Azure Container Instances group
- B. Use Azure Container Apps environments with multiple container apps (one per service) connected via the internal ingress/DNS
- C. Define all containers in a single ACI container group YAML file
- D. Use Azure App Service deployment slots for each container

**Correct Answer:** B  
**Explanation:** Azure Container Apps supports environments (the shared network boundary) with multiple Container Apps. Internal ingress allows container apps to communicate via DNS (e.g., http://api.internal.environment). Each service (frontend, API, Redis as a container) is a separate Container App within the same environment. This mirrors a microservices deployment on Kubernetes without cluster management.

## Skill Area: Develop AI solutions using Azure data management services (25–30%)

### Q3
**Question:** A developer is building a RAG chatbot that requires semantic vector search over a knowledge base. They need to store document embeddings (1536-dimensional vectors from text-embedding-ada-002) and query them by cosine similarity. Which Azure data service supports native vector search?

**Options:**
- A. Azure SQL Database with VECTOR data type (preview) or Azure AI Search vector fields
- B. Azure Table Storage with float array storage
- C. Azure Blob Storage with custom similarity computation
- D. Azure Cache for Redis (standard tier) with hash structures

**Correct Answer:** A  
**Explanation:** Azure AI Search supports vector fields with HNSW (Hierarchical Navigable Small World) indexing for approximate nearest neighbor (ANN) search at scale. Azure SQL Database now supports a VECTOR data type (in preview) with cosine similarity queries using DiskANN indexing. Both are valid architectures; Azure AI Search is the most mature option for production RAG at scale.

### Q4
**Question:** A developer needs to store AI application state including conversation history and user preferences in a NoSQL database that scales globally, supports point reads by user ID with single-digit millisecond latency, and is accessible via a Python SDK. Which Azure service is best suited?

**Options:**
- A. Azure SQL Database
- B. Azure Cosmos DB for NoSQL with the Python SDK
- C. Azure Table Storage
- D. Azure Database for PostgreSQL

**Correct Answer:** B  
**Explanation:** Azure Cosmos DB for NoSQL provides single-digit millisecond reads by partition key (user ID as partition key), global distribution, and flexible JSON document storage — ideal for conversation history and user state. The Python SDK (azure-cosmos) supports CRUD operations and change feed for real-time event processing. Cosmos DB Serverless is cost-effective for applications with bursty access patterns like AI chatbots.

### Q5
**Question:** A developer builds a RAG application that stores document chunks in Azure Database for PostgreSQL — Flexible Server. They need to perform vector similarity search on embedding columns. Which PostgreSQL extension enables this?

**Options:**
- A. pg_trgm (trigram similarity)
- B. pgvector extension for vector similarity search (cosine, L2, inner product)
- C. Full-Text Search (FTS) with tsvector
- D. PostGIS for spatial data

**Correct Answer:** B  
**Explanation:** The pgvector extension adds a `vector` data type and operators for L2 distance, inner product, and cosine distance to PostgreSQL. Combined with IVFFLAT or HNSW indexing, it enables efficient approximate nearest neighbor search directly in PostgreSQL — avoiding a separate vector database. Azure Database for PostgreSQL — Flexible Server supports pgvector natively.

## Skill Area: Connect to and consume Azure services (25–30%)

### Q6
**Question:** A developer is building an event-driven AI pipeline where document uploads to Azure Blob Storage should automatically trigger an Azure Function that processes the document with Azure AI Document Intelligence. What is the RECOMMENDED trigger configuration?

**Options:**
- A. Azure Function with HTTP trigger; client POSTs the blob URL on upload
- B. Azure Function with Azure Blob Storage trigger (event-based via Azure Event Grid)
- C. Azure Function with Timer trigger polling storage every minute
- D. Azure Logic Apps connector for blob storage

**Correct Answer:** B  
**Explanation:** The Azure Function Blob trigger (event-based, backed by Azure Event Grid) fires the function immediately when a new blob is uploaded. Event-driven triggers are more reliable and efficient than polling (timer). The function receives the blob name and container and can call Azure AI Document Intelligence with the blob URI. This is the standard serverless document processing pattern.

### Q7
**Question:** A developer needs to decouple an AI inference service from its clients. The service processes requests asynchronously: clients submit requests and poll for results. High throughput (10,000 requests/second peak) is required. Which Azure messaging service is MOST appropriate?

**Options:**
- A. Azure Service Bus Queue (FIFO, at-least-once, max 256 KB message)
- B. Azure Event Hubs (partitioned streaming, millions of events/second)
- C. Azure Storage Queue (simple, high throughput, 500 requests/second per queue)
- D. Azure Service Bus with sessions for ordered processing

**Correct Answer:** B  
**Explanation:** Azure Event Hubs is designed for high-throughput event streaming (millions of events per second) with partitioned consumers. For 10,000 requests/second, Event Hubs with multiple partitions handles the volume. Service Bus Queue is better for ordered, reliable message processing with complex routing. Storage Queue has lower throughput limits. The choice depends on ordering requirements: Event Hubs for high throughput streaming, Service Bus for reliable business process queues.

### Q8
**Question:** A developer builds an AI application that calls a backend Azure Function. They want to expose this function via a managed API with rate limiting, authentication, caching of responses, and request transformation without changing the function code. Which Azure service provides this?

**Options:**
- A. Azure Application Gateway
- B. Azure API Management (APIM)
- C. Azure Front Door
- D. Azure Traffic Manager

**Correct Answer:** B  
**Explanation:** Azure API Management (APIM) is the enterprise API gateway that sits in front of backend services (Functions, App Service, Logic Apps, etc.) and provides: authentication (OAuth 2.0, API keys, JWT validation), rate limiting and throttling policies, response caching, request/response transformation, subscription management, and a developer portal. No changes are needed to the backend function code.

## Skill Area: Secure, monitor, and troubleshoot Azure solutions (20–25%)

### Q9
**Question:** An AI application stores sensitive API keys (Azure OpenAI endpoint key, third-party API tokens) that are needed at runtime. What is the Azure best practice for managing these secrets?

**Options:**
- A. Store secrets in Azure App Configuration as plain text settings
- B. Store secrets in Azure Key Vault and use a Managed Identity to access them at runtime — no secret ever stored in code or config files
- C. Base64-encode the secrets and store them in environment variables
- D. Store secrets in Azure Storage as encrypted blobs

**Correct Answer:** B  
**Explanation:** Azure Key Vault is the standard service for storing secrets, certificates, and keys. Managed Identity (system-assigned on the App Service/Container App/Function) eliminates the need to manage credentials to access Key Vault — Azure AD handles authentication automatically. The application retrieves secrets using the Azure SDK (SecretClient) at startup or runtime. Never commit secrets to code or config files.

### Q10
**Question:** A developer needs to identify the root cause of intermittent latency spikes in an AI application that calls Azure OpenAI, Azure AI Search, and Azure Cosmos DB in a request pipeline. Which Azure Monitor feature provides distributed tracing across all these service calls?

**Options:**
- A. Azure Monitor Metrics dashboards for each service
- B. Application Insights — distributed tracing with the dependency tracking SDK
- C. Azure Log Analytics — separate KQL queries per service
- D. Azure Network Watcher packet capture

**Correct Answer:** B  
**Explanation:** Application Insights distributed tracing instruments each outbound call (Azure OpenAI, AI Search, Cosmos DB) as a dependency with duration and success metrics. The end-to-end transaction view (Application Map) shows the full call chain with timing for each hop, enabling root cause analysis of where latency originates. The Azure SDKs for OpenAI and Cosmos DB have built-in OpenTelemetry support that integrates with Application Insights.
